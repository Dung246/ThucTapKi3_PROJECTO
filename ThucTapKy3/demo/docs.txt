=== TOKEN & OUTPUT ECONOMY RULES ===
- Never run a command that can produce long, repetitive streaming output (e.g.
  `docker pull`, `docker compose up` when an image isn't cached yet, `docker logs -f`,
  verbose build tools) without redirecting/limiting it. Always pipe to `tail -n 20`
  (or similar) or use a quiet/no-progress flag, so only the final result appears in
  context.
- Before running `docker compose up` for any service, first run `docker pull <image>`
  separately in the background or confirm the image is already cached, so the compose
  command itself doesn't stream a full image-download log.
- When running diagnostic commands (docker ps, inspect, port checks, netstat, etc.),
  run the minimum number needed to identify the issue — don't run overlapping checks
  "just in case."
- When something fails, form a hypothesis first, then run one targeted command to
  confirm it, rather than running several exploratory commands in sequence.
- Summarize long tool outputs in your own words in the chat response instead of
  quoting them back in full.
- Prefer running background/long tasks with output written to a log file, then read
  only the last N lines, rather than streaming full output into the conversation.
- Keep this section itself concise — do not let it grow beyond ~15 lines.

  - Tools: Postman (API testing) — skeleton collection created session 3, Git/GitHub
- Architecture Summary:
  Three-tier architecture with three separate Docker containers:
  - frontend container: React application served by Nginx (3000:80)
  - backend container: Spring Boot REST API (8080:8080)
  - database container: MySQL 8 (3306:3306) with persistent volume

  Communication flow:
  ReactJS (Client)
      → HTTP/JSON
  Spring Boot REST API
      → JPA/Hibernate
  MySQL

  Authentication:
  JWT token in the Authorization header with Role-Based Access Control (RBAC).

  Run the entire system using:
  docker-compose up --build

=== API ENDPOINTS ===
(prefix /api. Implemented endpoints are marked [DONE]; the rest are still planned.)

- Auth
  - [DONE] POST /api/auth/register — public. Body: {fullName, email, password (min 8 chars — bumped from 6 on Day 10 to match the SRS field spec, see KEY DECISIONS), phone?}. Creates a CUSTOMER (password BCrypt-hashed). 201 + AuthResponse{token,id,fullName,email,role}; 409 if email already exists; 400 on validation failure.
  - [DONE] POST /api/auth/login — public. Body: {email, password}. 200 + AuthResponse (JWT, 24h expiry, claims: sub=userId, email, role); 401 invalid email/password; 403 if account status=LOCKED.
  - Public register() always creates a CUSTOMER; STAFF/ADMIN accounts are provisioned separately via POST /api/admin/staff (see Customer / User Management below, done as of Day 4).

- Category
  - [DONE] GET /api/categories — public, list all.
  - [DONE] GET /api/categories/{id} — public, 404 if not found.
  - [DONE] POST /api/categories — ADMIN only (401 no token, 403 wrong role). Body: {name, description?}. 201; 409 on duplicate name.
  - [DONE] PUT /api/categories/{id} — ADMIN only. Body: {name, description?}. 200; 404 not found; 409 duplicate name.
  - [DONE] DELETE /api/categories/{id} — ADMIN only. 204; 404 not found; 409 if the category still has products (see KEY DECISIONS).

- Product
  - [DONE] GET /api/products — public. Query params: name (partial, case-insensitive), categoryId, minPrice, maxPrice, page/size/sort (Spring Pageable, default size=10 sorted by id). Always filtered to status=ACTIVE (customer-facing). Returns a Spring Page<ProductResponse> (content, totalElements, totalPages, etc.).
  - [DONE] GET /api/products/{id} — public. 200 only if status=ACTIVE; 404 otherwise (matches UC-04 "product no longer exists" for hidden/soft-deleted products, including for Admin — see KEY DECISIONS).
  - [DONE] POST /api/products — ADMIN only. Body: {name, categoryId, price, quantity, imageUrl?, description?}. 201; 409 duplicate name; 404 if categoryId doesn't exist. `imageUrl` is a plain string, still accepted directly (see KEY DECISIONS, session 19) for seed/demo convenience or external URLs — a real image file goes through the new upload endpoint below instead.
  - [DONE] PUT /api/products/{id} — ADMIN only. Same body. 200; 404 not found or bad categoryId; 409 duplicate name.
  - [DONE] DELETE /api/products/{id} — ADMIN only. Always soft delete (sets status=INACTIVE, row is never removed). 204; 404 not found. Idempotent if already INACTIVE.
  - [DONE] POST /api/admin/products/{id}/image — ADMIN only (new, session 19, resolves the long-pending "product image storage" decision — see KEY DECISIONS). multipart/form-data, field name "file". Validates content-type starts with "image/" and size <= 5MB, 400 with a clear message otherwise (also 400 if the request exceeds Spring's own multipart limit, set generously above 5MB so this endpoint's own check is what normally fires). Saves the file under a local Docker named volume (product_images, mounted at /app/uploads in the backend container) with a generated UUID filename, sets that product's image_url to "/uploads/<uuid>.<ext>", and serves it back publicly via a static resource handler at GET /uploads/**. 200 + updated ProductResponse; 404 if the product doesn't exist. Works on any existing product, whether it already has an image_url or not — a second upload simply overwrites image_url with the new file's path (the old file is left on disk, not deleted).

- Cart
  - [DONE] GET /api/cart — CUSTOMER only. Returns CartResponse{items: [CartItemResponse], totalAmount}, scoped to the caller (from JWT).
  - [DONE] POST /api/cart — CUSTOMER only. Body: {productId, quantity}. Merges into the existing (user,product) row if present, else creates one; quantity capped at the product's current stock. 201; 404 if product missing/INACTIVE; 409 if product stock is 0.
  - [DONE] PUT /api/cart/{itemId} — CUSTOMER only, scoped to the caller's own items. Body: {quantity}. Capped at current stock. 200; 404 if the item doesn't exist or belongs to another user; 409 if stock is 0.
  - [DONE] DELETE /api/cart/{itemId} — CUSTOMER only, scoped to the caller's own items. 204; 404 if not found/not owned.

- Order
  - [DONE] POST /api/orders — CUSTOMER only. Body: {shippingAddress, paymentMethod: COD|BANK_TRANSFER}. Creates the order from the caller's current cart in one transaction: re-validates every cart line against live product status/stock, rejects the whole request (409, naming the unavailable products) if any line fails, else snapshots unit_price per line, deducts product stock, clears the cart, and records the initial PENDING row in order_status_history. 201 + OrderResponse; 400 if the cart is empty; 409 if any item is unavailable.
  - [DONE] GET /api/orders — any authenticated role. CUSTOMER gets only their own orders; STAFF/ADMIN get all orders, optionally filtered by `?status=`. Each entry includes line items but not the full status history (see KEY DECISIONS on list vs detail).
  - [DONE] GET /api/orders/{id} — any authenticated role. CUSTOMER: 404 if the order isn't theirs (not 403, consistent with the rest of the API); STAFF/ADMIN: any order. Includes line items and the full status_history audit trail.
  - [DONE] PATCH /api/orders/{id}/status — STAFF/ADMIN only. Body: {status, note?}. Enforces the forward-only state machine from Ban_hang_SRS.docx Hinh 2.4 (PENDING->{CONFIRMED,CANCELLED}, CONFIRMED->{SHIPPING,CANCELLED}, SHIPPING->{COMPLETED,CANCELLED}; COMPLETED/CANCELLED terminal). 200 + updated OrderResponse (with history); 404 not found; 409 if the transition isn't a valid edge (no skipping, no transitions out of a terminal state); 400 if transitioning SHIPPING->CANCELLED without a `note` (the diagram's "hiem, can ghi chu ly do" / delivery-failed edge). Special case (new, Day 10, UC-10): a COMPLETED->CANCELLED attempt is still always rejected (COMPLETED remains truly terminal - the order's status never actually changes to CANCELLED), but now 400s without a `note` and, when a note IS supplied, logs a new order_status_history row (oldStatus=newStatus=COMPLETED, the note as the reason) before responding 409 — "logs the rejected attempt for manual review" per UC-10's exception flow, instead of the old bare/unlogged 409.

- Admin Product Management
  - [DONE] GET /api/admin/products — ADMIN only (new, Day 8). Query params: name, categoryId, status (ACTIVE|INACTIVE, omit for both), page/size/sort. Same dynamic-Specification search as the public GET /api/products but WITHOUT the forced ACTIVE filter — this is how Admin browses/reactivates soft-deleted products, closing the Day 3 KNOWN ISSUE gap. Product create/update/soft-delete still go through the existing POST/PUT/DELETE /api/products(/{id}) (unchanged, still ADMIN only).

- Customer / User Management
  - [DONE] GET /api/admin/customers — ADMIN only. Lists role=CUSTOMER accounts.
  - [DONE] PATCH /api/admin/customers/{id}/status — ADMIN only. Body: {status: ACTIVE|LOCKED}. Implements UC-11; 404 if the target isn't a CUSTOMER account (also hides STAFF/ADMIN accounts from this endpoint).
  - [DONE] GET /api/admin/staff — ADMIN only. Lists role=STAFF or role=ADMIN accounts.
  - [DONE] POST /api/admin/staff — ADMIN only. Body: {fullName, email, password (min 8 chars), phone?, role: STAFF|ADMIN}. Delegates to AuthService.createAccount (same hashing/duplicate-email logic as public register). 201 + UserResponse; 409 duplicate email; 400 if role isn't STAFF/ADMIN.
  - [DONE] PUT /api/admin/staff/{id} — ADMIN only (new, Day 10, closes the UC-12 "sửa tài khoản" gap). Body: {fullName, email, phone?, role: STAFF|ADMIN} — no password field (no reset via this endpoint, out of scope). 200 + updated UserResponse; 404 if the target isn't a STAFF/ADMIN account; 409 if the new email belongs to a different existing account; 400 if role isn't STAFF/ADMIN.
  - STAFF is explicitly blocked from all five /api/admin/** endpoints (403) — per SRS 2.5.1 (and 2.5.2, corrected Day 10 to match — see KEY DECISIONS), only "Quan ly don hang" is shared between Staff and Admin; customer-lock and account-provisioning/editing are Admin-only.

- Review
  - [DONE] POST /api/products/{id}/reviews — CUSTOMER only. Body: {rating: 1-5, comment?}. UC-08 eligibility: the caller must have at least one order containing this product with status=COMPLETED. 201 + ReviewResponse; 403 if not eligible (never purchased/received it); 409 if the caller already reviewed this product (one review per product per customer — see KEY DECISIONS); 400 if rating is outside 1-5; 404 if the product id doesn't exist at all.
  - [DONE] GET /api/products/{id}/reviews — public. Newest-first list + averageRating + reviewCount. 200 (empty list if none); 404 if the product id doesn't exist (works for both ACTIVE and INACTIVE products — a discontinued product's historical reviews stay visible).

- Statistics
  - [DONE] GET /api/admin/statistics/revenue — ADMIN only (UC-13). Query params: granularity=DAY|MONTH|YEAR (default DAY), from/to (ISO dates, default all-time). Returns [{period, revenue}], computed only from status=COMPLETED orders, grouped by the chosen granularity. Empty array (200, not an error) if no COMPLETED orders fall in range.
  - [DONE] GET /api/admin/statistics/top-products — ADMIN only. Query params: from/to (default all-time), limit (default 10). Returns [{productId, productName, quantitySold, revenue}] ranked by quantity sold, computed from order_details joined to status=COMPLETED orders.

Session 5 (Day 2 implementation) decisions not previously covered in docs.txt:
- jjwt (io.jsonwebtoken) chosen for JWT generation/parsing over Spring Authorization Server or hand-rolled JWT. Why: it's the de facto standard for a simple stateless-JWT setup in Spring Boot, small API surface (JwtService is ~50 lines), no OAuth2/authorization-server complexity the SRS doesn't call for.
- Login/registration implemented as custom logic in AuthService (direct UserRepository lookup + PasswordEncoder.matches), not via Spring Security's AuthenticationManager/UserDetailsService/DaoAuthenticationProvider chain. Why: the SRS's auth model (email+password -> JWT with id+role claims) doesn't need Spring Security's session-oriented authentication abstractions; the custom path is far fewer moving parts to explain in the final report. Tradeoff: Spring Boot's UserDetailsServiceAutoConfiguration still fires and logs a "Using generated security password" warning at startup for an unused in-memory user — harmless (nothing uses it) but cosmetic noise; can be silenced later by excluding that auto-configuration if it bothers reviewers.
- Spring Security's `hasRole("ADMIN")` (i.e., authority `ROLE_ADMIN`) used for endpoint authorization, matching the SRS 2.5.1 permission matrix in Ban_hang_SRS.docx exactly: only Admin may create/update/delete categories ("Quản lý sản phẩm & danh mục" = X only for Admin); everyone (including anonymous) may read them, matching "Tìm kiếm & xem sản phẩm" = X for all three roles/guests in a typical storefront.
- GET /api/categories and GET /api/categories/{id} made `permitAll` (no auth required at all), not just "any authenticated role". Why: the SRS matrix marks product/category viewing as available to Customers browsing before login too (typical e-commerce flow: browse anonymously, log in only to buy) — there's no reason to force a login just to list categories.
- Category delete is blocked (409 Conflict) rather than cascading or soft-deleting when products still reference it. Why: categories.status doesn't exist in the schema (only products/users have a status column for soft-delete/lock), and products.category_id is NOT NULL, so silently cascading would either orphan or bulk-delete products — too destructive for a guarded action. Caller must reassign or remove the category's products first. This required adding a minimal ProductRepository (existsByCategoryId) a day early (see FILE/MODULE MAP).
- All API errors return a uniform JSON shape {timestamp, status, message} via GlobalExceptionHandler, rather than Spring Boot's default whitelabel error page or ad-hoc per-controller error handling. Why: consistent, predictable shape for the Postman tests and the future frontend to parse; one place to add new exception mappings as later days add more custom exceptions.
- Discovered mid-implementation that Spring Boot 4.1.0 ships Jackson 3.x, whose base package moved from `com.fasterxml.jackson.*` to `tools.jackson.*` (e.g. `tools.jackson.databind.ObjectMapper`) — this broke the first compile of the custom 401/403 JSON handlers. Fixed by importing the new package. Worth remembering for any future code that needs Jackson types directly (most Spring MVC code is unaffected since serialization is automatic).
- Postman collection's "Login - Locked Account" and "Delete Category - Blocked By Products" scenarios need one-time manual MySQL seeding (locking a user / inserting a product row) before they can pass, because no admin-lock API or Product-create API exists yet (Day 4 and Day 3 respectively). Both were verified correct via an equivalent manual curl+SQL script this session (all 15/15 TEST-02 scenarios passed) in addition to a partial newman run (see KNOWN ISSUES) — this is a temporary test-automation gap, not an application bug.

Session 6 (Day 3 implementation) decisions not previously covered in docs.txt:
- Product search/filter implemented with Spring Data JPA Specifications (JpaSpecificationExecutor), not a hand-written @Query or QueryDSL. Why: name/categoryId/minPrice/maxPrice are all optional and combine freely (any subset can be present) — Specifications let each filter be added conditionally (`spec.and(...)` only when the param is non-null) without a combinatorial explosion of derived-query methods or a giant dynamic JPQL string.
- Pagination uses Spring's built-in `Pageable`/`Page<T>` (query params: page, size, sort; default size=10, sorted by id) rather than a hand-rolled DTO. Why: zero extra code, standard Spring Boot convention the eventual React frontend can rely on directly (content/totalElements/totalPages/etc. are all present in the JSON as-is).
- GET /api/products and GET /api/products/{id} are both hard-filtered to status=ACTIVE for everyone, including Admin (no "show inactive too" mode/endpoint). Why: not explicitly requested for Day 3, and UC-04's exception flow ("San pham da bi an/xoa -> hien thi San pham khong con ton tai") doesn't distinguish Admin vs Customer. Consequence: after a soft delete, the only way to see/reactivate an INACTIVE product is a direct PUT by id (which doesn't go through the ACTIVE filter) or a DB query. Flagged as a possible Day 4+ gap if an "Admin product list including inactive" view turns out to be needed. (RESOLVED Day 8 — see KNOWN ISSUES: GET /api/admin/products was added specifically to close this gap.)
- DELETE /api/products/{id} always soft-deletes (sets INACTIVE), regardless of whether the product is actually referenced by any order — simpler than the SRS's literal wording ("only soft-delete if it's already in an order") and explicitly requested this way in today's task. Also matches SRS 3.3's stated end state either way (deleted-with-orders products must be INACTIVE, not gone) and costs nothing extra since Order API doesn't exist yet to hard-delete against anyway.
- Adding an out-of-stock product to the cart (product.quantity <= 0) is rejected with 409 (ProductOutOfStockException) rather than silently creating/leaving a quantity=0 cart line. Why: SRS UC-06 already treats "product out of stock" as an explicit exception case at checkout ("remove item from cart"); rejecting earlier, at add/update time, avoids ever having a nonsensical zero-quantity cart_items row. Everywhere else the exact SRS rule applies unchanged: requesting more than available stock silently caps at current stock (UC-05), it does not reject.
- Cart cross-user protection returns 404 (not 403) for another user's cart item id. Why: consistent with the existing CategoryNotFoundException/ProductNotFoundException pattern, and avoids confirming to a caller that an item with that id exists at all (better security posture than a 403, which would leak existence). All cart repository lookups are scoped by (itemId, userId) together so this happens naturally, not via a manual ownership-check-then-403 branch.
- Cart line pricing is computed live from the product's current price at request time (CartItemResponse.unitPrice/lineTotal), not stored/snapshotted on the cart_items row. Why: matches the DB schema exactly (cart_items has no unit_price column, unlike order_details which does) — the schema's intent is clearly that price gets fixed/snapshotted only at order creation (Day 4), not in the cart.
- @AuthenticationPrincipal Long userId used directly in CartController to get the current user's id. Why: JwtAuthenticationFilter already sets the Authentication's principal to the raw userId (Long) rather than a UserDetails object (see Day 2 decisions on skipping Spring Security's UserDetailsService chain), so Spring's default AuthenticationPrincipalArgumentResolver resolves it with zero extra code.
- Product/Cart required no new Gradle dependencies or application.properties changes — everything needed (Spring Data JPA Specifications, Bean Validation, Spring Security RBAC) was already added in Day 1/2.

Session 7 (Day 4 implementation) decisions not previously covered in docs.txt:
- The order status state diagram (Ban_hang_SRS.docx Hinh 2.4) is an embedded image, not extractable as text by the docx->plain-text pipeline used in earlier sessions. Rather than guess the edges from the surrounding prose, unzipped the docx and viewed the actual PNG (word/media/d77f8b48....png, matched to "Hinh 2.4" by relationship-ID order in document.xml.rels) directly. Confirmed exact edges: PENDING->{CONFIRMED, CANCELLED}, CONFIRMED->{SHIPPING, CANCELLED}, SHIPPING->{COMPLETED, CANCELLED}; COMPLETED and CANCELLED are terminal (no outgoing edges). The SHIPPING->CANCELLED edge is specifically labeled "Giao that bai (hiem, can ghi chu ly do)" (delivery failed, rare, needs a reason) — the only edge in the diagram that calls for a note, so that's the only transition where `note` is enforced as required (400 if missing). This is worth remembering: when the docx mentions a diagram/wireframe by figure number in future days (Review/Statistics wireframes etc.), check word/_rels/document.xml.rels for the rId order (matches document order 1:1 in this file) to find the right image rather than guessing from prose alone.
- Order creation's stock re-check rejects a cart line if `product.quantity < cartItem.quantity` (not just `== 0`). Why: the task/SRS literally only calls out "stock is now 0," but only checking for exactly 0 would still let an order oversell if stock dropped to, say, 1 while a cart held a quantity of 3 — checking "enough stock for the requested quantity" is the correct generalization and costs nothing extra.
- Order creation is entirely a customer self-service action from their own cart; there is no endpoint to create an order on a customer's behalf or from an arbitrary product list. Matches UC-06 exactly (order = current cart -> checkout).
- Chose a dedicated order_status_history table over an audit column on `orders`. Why: a status can transition multiple times (PENDING->CONFIRMED->SHIPPING->COMPLETED is 3 transitions plus the initial creation = 4 rows) and a single column/pair of columns can't represent that without concatenation hacks; a table is the standard, queryable way to do an audit trail and directly satisfies UC-10's "audit trail...phuc vu tra soat" (serve investigation/review) wording by being retrievable, not just logged to a file. Exposed via GET /api/orders/{id}'s statusHistory array rather than a separate endpoint, since it's cheap (one extra query) and directly useful without adding new routes.
- `note` is required by the service layer only for the SHIPPING->CANCELLED transition (not for PENDING->CANCELLED or CONFIRMED->CANCELLED), matching the diagram literally (only that edge is annotated as needing a reason). The SRS's UC-10 exception text ("bat buoc ghi chu khi tu choi huy don hoac xu ly ngoai le") is slightly broader in prose, but since a rejected/invalid transition never actually changes state (it 409s before any note would matter), the only case where a note is meaningfully "required" is a transition that's exceptional but still allowed — which is exactly the SHIPPING->CANCELLED edge (and, as of Day 10, COMPLETED->CANCELLED rejected attempts — see KEY DECISIONS Session 13).
- No customer self-service cancel endpoint was built — today's task explicitly scoped PATCH .../status to STAFF/ADMIN only, even though the diagram labels PENDING->CANCELLED as "Khach huy / het hang" (customer cancels / out of stock), which reads as a customer-initiated action. For now, a customer wanting to cancel talks to support/staff, who perform the transition on their behalf; a dedicated customer-facing cancel endpoint is flagged in KNOWN ISSUES as a likely future addition (would need its own narrower rule: only from PENDING, only the order's own owner).
- GET /api/orders (list) omits the full statusHistory per order (uses OrderResponse.summary(), empty history array) while GET /api/orders/{id} includes it (OrderResponse.detail()) — avoids an extra history query per row in list views. Line items are included in both, since an order without knowing what's in it isn't very useful even in a list (e.g. for Staff picking which order to process). No JOIN FETCH / N+1 optimization was done for the list endpoint; acceptable at this project's scale (course demo, tiny datasets) but worth revisiting if it were ever a real production system.
- GET /api/orders/{id} returns 404 (not 403) when a CUSTOMER requests another customer's order — same pattern as Cart/Product/Category from earlier days (don't confirm existence of a resource you can't access).
- SecurityUtils (currentUserId/currentRole from Authentication) added as a small static-method utility rather than repeating role-extraction logic inline in OrderController, or introducing a custom @CurrentUser annotation/argument resolver. Why: OrderController is the first controller that needs both the id and the role (Cart only ever needed id via @AuthenticationPrincipal); a 15-line utility class was the least amount of new machinery to get there, and it's reusable if a future controller needs the same thing.
- AuthService.createAccount() extracted as a reusable method (register() and AdminService.createStaffAccount() both call it, only the `role` argument differs) per the task's explicit instruction to avoid duplicating hashing/duplicate-email logic. This is the project's first cross-service reuse of a service method (AdminService depends on AuthService directly, not through an interface) — acceptable at this scale; would introduce an interface/facade only if a third consumer showed up.
- Admin's customer-lock endpoint (PATCH /api/admin/customers/{id}/status) is scoped to role=CUSTOMER only (filters the lookup, 404 if the target is STAFF/ADMIN) rather than allowing Admin to lock any account through it. Why: the endpoint's whole purpose (UC-11) is customer account moderation; letting it also lock STAFF/ADMIN accounts would silently create an undocumented "lock any user" capability the SRS never asked for, and there's no requirement today for staff/admin account locking.
- POST /api/admin/staff validates `role` must be STAFF or ADMIN (400 otherwise, rejecting CUSTOMER) via a manual check rather than a Bean Validation `@AssertTrue`/custom-annotation, since Role has a third value (CUSTOMER) that Bean Validation can't easily exclude from a plain `@NotNull` on an enum field — simplest to just check it in the service.

Session 8 (Day 5 implementation) decisions not previously covered in docs.txt:
- One review per (product, customer) ever, not one per completed order. Why: UC-08 describes reviews as being about the product itself ("danh gia san pham"), not a specific order line, and reviews are publicly aggregated into a single average rating per product — letting a repeat buyer submit multiple reviews for the same product would let one customer's opinion skew that average multiple times over, which isn't the intent of a product rating. Enforced at the DB layer too (UNIQUE(user_id, product_id) on reviews), consistent with the project's established pattern of pairing a DB constraint with a service-level check (see Day 1's products.name/categories.name decision). There's no review-update/delete endpoint, so a customer who wants to change their rating currently can't — not required by UC-08 and flagged in KNOWN ISSUES rather than built speculatively.
- Review-not-eligible returns 403 (Forbidden), while a duplicate review returns 409 (Conflict) — different codes for what might look like similar "can't do this" cases. Why: eligibility is about the caller's authorization state (they haven't earned the right yet, a 403-shaped problem, same family as AccountLockedException elsewhere in this project), whereas "already reviewed" is a conflict with existing data (same family as DuplicateEmailException/DuplicateProductNameException/DuplicateCategoryNameException, all 409). Consistent with how those exact distinctions were already made in Days 1-4.
- GET /api/products/{id}/reviews does not require the product to be status=ACTIVE (unlike GET /api/products/{id} itself, which 404s on INACTIVE — see Day 3's decision). Why: a product's reviews are historical record; hiding them the moment a product is discontinued would be actively wrong (the reviews are still true and useful), whereas the Day 3 decision about hiding the product page itself was about not selling something that's no longer available — different concerns, so the two endpoints intentionally disagree on ACTIVE-filtering.
- ProductReviewsResponse computes averageRating in Java (a stream over the already-fetched review list) rather than a separate SQL AVG() query. Why: the endpoint already has to fetch the full review list to return it, so a second round-trip just for the average would be pure waste at this project's scale; would reconsider only if the review list itself became paginated (not requested today).
- Statistics queries use native SQL (@Query(nativeQuery = true)) rather than JPQL or Specifications. Why: MySQL's DATE_FORMAT()/GROUP BY-by-formatted-date and the top-products JOIN+aggregate+LIMIT pattern don't map cleanly onto JPQL (no portable date-truncation function, and JPQL's GROUP BY story is awkward for this shape); since the DB is already locked to MySQL for this project (see Day 1 decisions), a native query is simpler and more readable than fighting JPQL for no portability benefit that will ever be used.
- RevenueGranularity (DAY/MONTH/YEAR) modeled as an enum carrying its own MySQL DATE_FORMAT pattern string as a field, rather than a switch statement in the service mapping a plain enum to a pattern. Why: keeps the granularity->pattern mapping colocated with the enum values themselves (one place to look, one place to add a new granularity later) instead of splitting that knowledge between the enum definition and a separate switch elsewhere.
- Statistics date-range params (from/to) default to "all time" (1970-01-01 through today) when omitted, rather than a rolling window like "last 30 days". Why: UC-13's own exception flow treats "no data in range" as a valid, non-error result ("Khong co du lieu"), so there's no need to guess a default window that might just produce an empty/confusing result for a demo dataset seeded across a handful of test days — all-time is the least surprising default and always shows something if any COMPLETED orders exist.
- Object[] rows from the native queries are converted to DTO fields via defensive `((Number) row[i]).longValue()`/toBigDecimal() casts rather than assuming a specific JDBC type. Why: MySQL Connector/J's exact return type for SUM()/COUNT() over INT/DECIMAL columns isn't something to hardcode a cast to (could come back as Long, BigInteger, or BigDecimal depending on driver version/column type) — casting via the common Number supertype is defensive and cheap.
- Backend Dockerfile uses a copy-dependencies-then-copy-src layering split (COPY gradlew/build.gradle/settings.gradle/gradle first, run a dependency-only step, then COPY src and run the real build) so that changing only application code doesn't force Gradle to re-resolve every dependency from Maven Central on every image rebuild — standard Docker layer-caching practice for Gradle/Maven projects, meaningfully speeds up iterative rebuilds once the dependency layer is cached.
- Chose environment-variable overrides (SPRING_DATASOURCE_URL/USERNAME/PASSWORD set directly in docker-compose.yml's `backend` service) over adding a Spring `application-docker.properties` profile file. Why: Spring Boot's environment variables already take precedence over application.properties with zero extra Spring config, so this needed no new file, no `SPRING_PROFILES_ACTIVE` wiring, and can't drift out of sync with a second properties file the way a profile file could; the tradeoff (config split between a .properties file and docker-compose.yml instead of centralized in one place) is minor at this project's scale. The task's instructions explicitly offered either approach.
- `depends_on: mysql: condition: service_healthy` used (not a bare `depends_on: [mysql]`) so Compose actually waits for MySQL's healthcheck to pass before starting the backend container, not just for the mysql container process to exist — verified this matters: MySQL takes a few seconds after container-start before it's ready to accept connections, and the backend would otherwise race it on a cold `docker compose up`.
- Backend container is left running after this session for the first time (previously only sales_mysql was left up) — the task explicitly asked for the stack to be verified end-to-end and left in that state. To stop both: `docker compose down` (same command as before, now also stops sales_backend). To rebuild after a code change: `docker compose up --build -d`.

Why: Tailwind's utility-class approach maps directly onto the wireframes' simple, consistent visual language (navy header, white cards, minimal chrome) without needing to fight a component library's own opinions the way Bootstrap's pre-styled components would; v4's @tailwindcss/vite plugin needs zero config file for a project this size (no tailwind.config.js required, just `@import "tailwindcss";` in index.css), which is less setup than Bootstrap + any React wrapper (react-bootstrap) would need. Why: this is a course project with no XSS-hardening requirement in the SRS's non-functional requirements, and localStorage is dramatically simpler to wire up with a plain Bearer-token Axios interceptor than cookie-based auth would be (which would need CSRF protection, SameSite tuning, and backend changes to issue Set-Cookie) — acceptable tradeoff at this project's scale, flagged in KNOWN ISSUES as a security simplification for the record.

- Discovered and fixed a real backend gap during browser testing: SecurityConfig had no CORS configuration at all. Every one of Days 1-5's testing (curl, Postman/newman) was same-origin-equivalent from the browser's perspective (curl doesn't enforce or even know about CORS), so this was invisible until an actual browser tried to call the API from a different origin (localhost:3000 -> localhost:8080) and got silently blocked. This is a good reminder for future backend work: curl-based/Postman-based API testing cannot catch CORS issues, only a real browser can — worth remembering if more browser-only bugs surface in Day 7/8's fuller UI testing.
- Product list filters apply on explicit form submission ("Ap dung" button), not live-typing with a debounce. Why: simpler to implement correctly (no debounce timer/cleanup logic, no risk of a request race between fast keystrokes) and matches the wireframe's static form fields; a live-search UX polish could be added later but isn't requested by the SRS or today's task.
- Product prices are formatted and displayed as VND (₫) via `Intl.NumberFormat("vi-VN", {style:"currency", currency:"VND"})`, matching the Vietnamese-market SRS. Why this needed deciding today: the `price` column has no currency field or unit documented anywhere in DATABASE SCHEMA or prior KEY DECISIONS — earlier days' Postman/curl test data used arbitrary small decimal values (e.g. 999.99, 50.00) that were never rendered through a currency formatter, so the ambiguity never surfaced until the frontend had to actually display a price. Decided: `price` is a whole-VND amount (no meaningful subunits in real VND), so test/demo data going forward should use realistic VND-scale integers (e.g. 29990000 for a phone), not USD-scale decimals — the first browser test caught this exact mismatch (999.99 rendered as a nonsensical "1.000 ₫" after VND rounding) and the demo seed data was corrected accordingly.
- No chromium-cli or project-specific "run" skill existed for this repo yet, so Playwright + a Chromium binary were installed ad hoc (via npm, in the scratchpad, not added to the frontend's own package.json/devDependencies) purely to drive the golden-path browser test described in requirement 6. This tooling is not part of the shipped project and was not made persistent/repeatable as a project skill — a future session doing heavier UI testing (Day 7's "UI flow testing" task) should consider running `/run-skill-generator` to capture a proper project-level run/test skill instead of re-improvising this each time.

Accepted tradeoff: if the second call fails (e.g. a flaky network blip right after a successful create), the product exists without its image and the admin has to retry via Edit - not transactional, but low-stakes for a course project's admin-only, low-frequency operation, and the existing Edit flow already provides the retry path with zero extra code.
- File validation limits: content-type must start with "image/" (matches the task's literal wording, not a stricter fixed allow-list of jpeg/png/gif/webp) and size <= 5MB, chosen as a reasonable product-photo ceiling with no specific requirement driving the exact number. Both checks live in ProductService.uploadImage(), not a `@Valid` Bean Validation annotation on a DTO, since `MultipartFile` isn't a request-body DTO field the way every other validated input in this project is - checked manually near the top of the method instead, same emptiness-then-shape-then-business-rule check ordering used elsewhere (e.g. AuthService).
- Filename collisions handled by generating `UUID.randomUUID() + originalExtension` server-side rather than trusting the client's original filename - two admins uploading two different files both named "photo.jpg" get two different files on disk with zero chance of collision or one silently overwriting the other, and the extension is preserved so the file still opens/renders correctly by type.
- A second upload to the same product overwrites image_url with the new file's path but does NOT delete the old file from disk - simplest correct behavior (no risk of deleting a file something else still references) at the cost of accumulating orphaned files over time. Flagged under KNOWN ISSUES as an accepted, low-stakes cost rather than solved with a cleanup job that wasn't asked for.
- Frontend still shows the "Ảnh (URL)" text field alongside the new file input (not replaced) so both entry paths from the decision above are actually reachable from the UI, not just the API. If both are present when saving, the file upload wins (it runs second, after the JSON create/update already set whatever imageUrl the text field held) - documented directly in the form's own inline help text ("dùng cho dữ liệu mẫu" / "for sample data") so this priority order isn't a hidden surprise.
